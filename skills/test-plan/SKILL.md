# anti-legacy:test-plan

Produces the **detailed, human-facing functional Test Strategy** that stakeholders
read to understand how the modernized system will be proven correct. It is a
**deliverable** in the §"deliverables" suite: it RENDERS deterministically from the
pipeline's structured data — it does not coin prose with an LLM, and it does not
advance any phase or gate.

## Mental model — what this is, and what it is NOT

This deliverable is **distinct** from `anti-legacy:test-strategy`:

| | `anti-legacy:test-strategy` (pipeline-internal) | `anti-legacy:test-plan` (this deliverable) |
|---|---|---|
| Output | `contracts/{domain}/{req_id}.contract.json` + the master `.anti-legacy/contracts/test-strategy.md` | `.anti-legacy/deliverables/test-strategy.md` |
| Audience | the build / functional-tests engine | stakeholders, QA leads, reviewers |
| Content | per-requirement scenarios + parity rules (the executable contract) | the *functional test approach* across four test types, the parity safety net, environments, data, and the traceability matrix |
| Phase | a mainline phase that advances | **register only — never advances** |

The document itself states this distinction at the top so no reader confuses the
two `test-strategy.md` files.

The plan is **functional, not unit**. It covers four test types chosen for
modernization safety:

1. **Data-parity / equivalence** — legacy golden output vs. modern output on
   identical inputs. The primary correctness net for a migration.
2. **UAT acceptance (Given/When/Then)** — business-readable Gherkin scenarios a
   stakeholder signs off; feeds GATE_4_UAT.
3. **End-to-end business journeys** — a real flow crossing capabilities in
   dependency order, asserting the end state.
4. **API / contract** — request/response schema, status codes, error envelopes.

## Cross-Platform Notes

Pure standard-library Python (`python3`), every path built with `os.path`, no
shell-isms. Runs identically on macOS / Linux / WSL / Windows. It imports the
shared `antilegacy_core.deliverables` library (the dispatcher puts it on
`PYTHONPATH`) for all loading, writing, and registration — it never re-implements
loaders and never anchors on `__file__`.

## When it runs & prerequisites

Runs **"when the graph is ready"** — once
`.anti-legacy/requirements/requirements_graph.json` exists. It is richest after
`anti-legacy:test-strategy` has produced per-requirement contracts and
`anti-legacy:blueprint` has produced entity schemas, but it **degrades gracefully**:

- **Contracts absent** (test-strategy not run yet) → the plan is rendered from the
  graph + parity-relevant entity fields, and the document clearly notes
  "per-requirement contracts not yet generated by `anti-legacy:test-strategy`".
- **Blueprint absent** → the parity table falls back to numeric entity fields from
  the requirements graph; API in-scope notes degrade.
- **`deployment_target` absent in config** → the production rung of the
  environment ladder is rendered as a placeholder, not invented.

## Parameters

This skill takes no required parameters. The script flags (all optional, defaulting
to the standard `.anti-legacy/...` paths via the library):

- `--requirements` — path to `requirements_graph.json`.
- `--contracts` — path to the `contracts/` directory.
- `--blueprint` — path to `blueprint.json`.
- `--config` — path to `config.json`.
- `--no-register` — write the document but do not touch the manifest (hermetic
  dry run / preview).

## Step 1: Verify the requirements graph exists

```bash
python3 .anti-legacy/run.py manifest status
```

You want `requirements-graph` registered. Contracts (`test-strategy`) and
`blueprint-json` being present makes the plan richer but are not required — the
plan names what is missing rather than failing.

## Step 2: Render the test strategy

Run the renderer through the dispatcher. With no flags it reads the standard paths
and writes `.anti-legacy/deliverables/test-strategy.md`, registering the artifact:

```bash
python3 .anti-legacy/run.py test_plan
```

Preview without touching the manifest:

```bash
python3 .anti-legacy/run.py test_plan --no-register
```

Override any input when needed:

```bash
python3 .anti-legacy/run.py test_plan \
  --requirements .anti-legacy/requirements/requirements_graph.json \
  --contracts .anti-legacy/contracts \
  --blueprint .anti-legacy/requirements/blueprint.json \
  --config .anti-legacy/config.json
```

## Step 3: Read what was produced and sanity-check it

Open `.anti-legacy/deliverables/test-strategy.md` and confirm:

- **Overview** names the project, target stack, the four test types, and states
  explicitly that this is functional, not unit, testing.
- **One section per test type** has an approach, in-scope statement, entry/exit
  criteria, and a tooling note keyed to your `config.target_stack` (java →
  JUnit 5, + REST-assured for API; python → pytest; UAT → Gherkin; data-parity →
  golden legacy-vs-modern; E2E → journeys across domains in dependency order).
- **Data-parity safety net** enumerates every parity-sensitive output — contract
  `parity_rules` (field / precision / source_type) AND numeric / packed-decimal
  entity fields from the blueprint and graph — and calls out COMP-3 silent
  precision-loss risk. If it reads "_No parity-sensitive outputs found_", that is
  the document telling you the entity schema lacks precision metadata or contracts
  are not yet generated — fix upstream, do not hand-edit.
- **Traceability matrix** lists `req_id → domain → legacy_components → covering
  test types → contract scenario ids`, and any requirement with no contract is
  flagged in a dedicated **Coverage gaps** subsection.
- **Coverage summary** reports # requirements, # with contracts, # parity rules,
  and lists dropped (out-of-scope) requirements explicitly.

## Step 4: Confirm registration

```bash
python3 .anti-legacy/run.py manifest check deliverable-test-strategy
python3 .anti-legacy/run.py manifest status | grep deliverable-test-strategy
```

`manifest check` resolves the stored path and re-checksums it; a clean result
means the registered artifact matches the file on disk.

## Done-gate (BLOCKING — assert BEFORE registering)

The script enforces this itself and exits non-zero if it fails:

- `requirements_graph.json` exists and has at least one domain (else there is no
  requirement set to plan against — hard error, nothing is written).
- The rendered `.anti-legacy/deliverables/test-strategy.md` is **non-empty**.

On success the script prints the written path; with registration on it also prints
the registered artifact id and stored path. If the done-gate fails, the script
surfaces the specific gap on stderr and writes nothing — do not work around it.

## Output

- `.anti-legacy/deliverables/test-strategy.md` — the functional test strategy
  document (DISTINCT from the internal `.anti-legacy/contracts/test-strategy.md`).
- Manifest artifact `deliverable-test-strategy` (`format: markdown`,
  `produced_by: anti-legacy:test-plan`, `depends_on:
  ["requirements-graph", "test-strategy"]`, `status: final`), plus an
  `anti-legacy:artifact-registered` audit row. **The phase is never advanced.**

## Failure cases

- **"requirements graph not found or has no domains"** — the graph has not been
  built. Run the pipeline through `anti-legacy:graph-translator` first.
- **Sparse parity table / "No parity-sensitive outputs found"** — no contract
  declared `parity_rules` and no entity field carries a numeric / packed-decimal
  `source_type`. Run `anti-legacy:test-strategy` (for contract parity rules) and/or
  enrich the blueprint entity schema with `source_type`, then re-run.
- **Every requirement shows "(no contract)" in the matrix** — `anti-legacy:test-strategy`
  has not run yet. The plan is still valid (graph-derived); generate contracts and
  re-render to populate coverage.
- **`manifest check` reports drift** — the doc was hand-edited after registration.
  Re-run `test_plan` to regenerate and re-register rather than patching by hand.
