Why legacy modernization fails

Nobody knows what
the old system
actually does.

Legacy systems — mainframes running COBOL, enterprise Java monoliths, SAP customizations — accumulate decades of business logic with no documentation and no tests. Modernization projects fail not because the engineers are bad, but because the requirements are buried in the code itself.

×
No tests to verify against.

The old system has no test suite. You can't check whether your new version does the same thing.

×
No documentation to read from.

The spec is out of date. The business rules changed. The person who knew retired in 2009.

×
Rewrite and hope is not a plan.

Translating code line-by-line preserves the implementation, not the behavior — and gets the wrong bugs.

The big idea

Map first.
Build second.
The graph is
the proof.

anti-legacy separates the question "what does the old system do?" from the question "how should the new system do it?" — and answers the first one completely before starting on the second.

It builds two graphs. They are deliberately distinct artifacts.

Code graph · amber · evidence

The legacy estate, indexed and annotated. Every module, paragraph, dataset, table — with a business rule written onto each behavior-bearing node. This is a map of what the old code does. No guesses; every claim carries a confidence score and a source-line reference.

Domain graph · cyan · plan

The reimagined target, organized by business capability — not by what the old code happened to do. This is the blueprint for what the new system will do. It must cover every requirement from the code graph. Zero silent drops.

code node ▸ rule ▸ requirement ▸ code

That chain is unbreakable. A code change without a matching rule update fails CI.

01 — INDEX

One unified code graph

Index the estate
into a single
code graph.

The first step is structural, not semantic — it reads the code as a topology, not as logic. Every module, paragraph, dataset, table, transaction — captured as typed nodes with edges carrying {confidence, provenance}. This is the evidence spine. Across all source repos, unified.

Supports COBOL / JCL / CICS / IMS / DB2 mainframe estates and modern languages — Java, C#, Python, TypeScript, Go, and more — in a single indexing pass.

  • MOD Program / module — the orchestration spine.
  • PARA Paragraph / method — a behavior-bearing unit.
  • DS Dataset / file — persistent data source.
  • DB Database table — relational source of record.

02 — ANNOTATE

Adaptive ring expansion

Write the
business rule
onto every node.

Agents crawl the graph with adaptive ring expansion — starting at a node, then expanding outward one hop at a time until there is enough context to state the rule confidently. Each behavior-bearing node ends resolved (rule + confidence + provenance) or risk-flagged for a human to research.

Rules are written into the graph's native fields and mirrored to an append-only annotations.jsonl sidecar — the tamper-evident audit trail.

A resolved annotation · written to the graph

03 — COVER

A provable terminal condition

Resolved or flagged.
Zero unaccounted.

Coverage is not a percentage you eyeball. It is a terminal condition: every behavior-bearing node must be resolved (rule written, confident) or flagged (explicit human research needed). Zero nodes unaccounted for, zero silent gaps.

The flags are the HITL research queue — not failures, not blockers. The resolved slice can build immediately.

accounted · resolved + flagged = 100%

Example coverage ledger

resolved · rule written flagged · HITL queue

The pipeline exits this phase only when the ledger is full — every behavior-bearing node in one of the two states. No node can fall through.

04 — HITL

A queue, not a stop sign

Answer the risks.
Never block
resolvable work.

Human-in-the-loop (HITL) is not a gate that halts the whole pipeline. It is a targeted queue: the nodes an agent couldn't resolve with sufficient confidence, passed to a human for research. The resolved slice builds immediately. Only risk-touching work waits for an answer.

A two-layer gate — mechanical check plus a clarify review — closes each flag. When a human answers a risk, the node flips resolved and rejoins the build lane.

05 — RE-THINK

A new artifact — the plan, not an overlay

The domain graph —
by capability,
not by module.

The annotated rules are rationalized into a new target-state domain graph — organized by what the business needs, not by the shape of the old code. When migrating from multiple source repos, rules are merged and conflicts surfaced explicitly. The domain graph is a re-thought artifact — but it must cover every requirement edge from the code graph. Zero silent drops.

every original requirement covered · 0 dropped
06 — BUILD + TEST

Parity proven per requirement

The round-trip
proof.

A swarm of subagents builds incrementally against the domain graph, one requirement at a time. Parity is proven per requirement — test contracts pin the exact expected outputs so the modern service matches legacy behavior exactly, not approximately. For financial systems: decimal precision, rounding modes, error codes — all specified and verified.

07 — GATES + DRIFT

Real evidence · keep the graph true

Keep the graph
true forever.

Five gates span the pipeline: design, plan, build integrity (automated), semantic round-trip review, and UAT. Human gates require a real evaluator — no rubber-stamps, no phantom-green. The keystone is the CI drift gate: if code changes without a matching rule update, CI fails. The graph can never silently drift from the code it describes.
